# Architecture Decision Records

This directory holds the ADRs for radio-server. Each ADR is a short, dated record of one significant
decision and its context (ADR-first is the project convention — see [CLAUDE.md](../../CLAUDE.md)). The
table below is the index: what each ADR decided, its current status, and any supersession. It is
maintained by hand — add a row when you add an ADR.

## Read this first — two things the numbering can't tell you

- **`0001` is used twice.** [`0001-cycle-model.md`](0001-cycle-model.md) is a *process/meta* ADR (the
  headless-cycle working model, no `Status` line); [`0001-two-backend-radio-abstraction.md`](0001-two-backend-radio-abstraction.md)
  is the first *technical* ADR. They are kept as-is rather than renumbered, so existing references stay
  valid; a future tidy could renumber the process ADR to `0000`.
- **`0035` and `0036` are missing, and an earlier `0049`–`0053` never existed on disk.** Those numbers
  belonged to a reverted **M17 / mrefd / Codec2** linking arc that was rolled back wholesale (commit
  `f39a750`, "Revert cycles 41-58"). That arc is **dead**. Its live successor is the **Mumble/Murmur
  link** (ADRs 0041→0049). The current `0049` below is a completely different, live ADR that happens to
  reuse the number. **Anything in the tree referencing M17, mrefd, Codec2, libcodec2, or reflectors is
  stale and does not describe the shipped system.**

| [0090](0090-tx-timeout-timer.md) | **Transmitter time-out timer: a hard cap on continuous key, on every path.** The ADR 0089 folded crossband could get **stuck keying the transmitter** (dead air on the air) with no path to drop PTT but a process kill — because every existing safeguard (`tx.idle_timeout`, D-STAR/Mumble `tx_hang`) is a *silence* timeout that resets per frame, so a **continuous** key is uncapped. | [0091](0091-dstar-crossband-safety.md) | **D-STAR crossband safety: the over always closes, PTT always drops, the reflector is signal-gated.** The ADR 0089 folded crossband could get **stuck keying the transmitter** — inbound DATA kept arriving (queue never idled) while decodes stopped feeding RF, so `_reflector_to_rf`'s only close paths (end-bit / queue-idle) never fired; and `_teardown` relied on that loop's `finally`, which can't run while the task is **parked in a blocking vocoder decode** (cancel undeliverable, join hangs, and the vocoder that would unblock it was closed *after* the join). Only a process kill freed PTT. Five `dstar/bridge.py` fixes: (1) `_teardown` closes the vocoder **first** (unblocking a parked `_exchange`), force-drops PTT **directly** via new `_force_unkey`, and **bounds** each task join; (2) an **independent RX watchdog** closes the over when the keying session is idle past `tx_hang` (`TxSession.idle_elapsed`) even under continuous inbound; (3) a `write_timeout` on the DV Dongle serial port so a stuck write can't park forever; (4) `_rf_to_reflector` is **signal-gated by a per-link `AudioLevelGate`** (reusing the `audio.vad_*` thresholds, injected by `build_app`) so the crossband keys the reflector only on real RF audio — independent of the global `audio.squelch`; (5) `TxSlot` release only if actually held (never freeing a browser talker's slot) + a `_reap_stale_tx` backstop for a leaked "op" over. No new config, no canary change (gate reuses `audio.vad_*`; DI seam stays ungated). Sits above the ADR 0090 TOT. Verified on fakes (idle-watchdog close under failing decodes; `stop()` during a blocked decode drops PTT and returns; gate keys only on signal; slot accounting; op-over reap) — 1223 passed; re-enable gated on a bench proof (dummy-load reflector→RF + `doctor --dstar-browser-echo`) | Accepted |
| [0092](0092-dstar-parked-decode-ptt-safety.md) | **The parked decode can no longer hold PTT: an independent watchdog, a re-key guard, an unconditional unkey.** ADR 0091 was verified on fakes and merged — then the first **dummy-load** reflector→RF test on the real DV Dongle stuck-keyed anyway (`mode=rx`, PTT held ~40 s to the TOT). Two hardware realities the FakeVocoder/MockRadio couldn't reproduce: (1) ADR 0091's idle watchdog lived at the **top of the `_reflector_to_rf` loop**, but that loop `await`s each decode in the single-worker executor and a **wedged DV Dongle decode parks the loop there** — so the loop-top check never runs (`asyncio.wait_for` around `run_in_executor` doesn't help: it awaits the uncancellable thread to finish anyway); (2) `unlink` returned but PTT stayed up — `TxSession.close()` transmits a Part-97 sign-off **ID before** `ptt(False)`, that transmit **raised** on the wedged backend, and `_force_unkey`'s `suppress` swallowed it, skipping the unkey; and a decode resuming after the over closed could **re-key**. Four fixes, no config/canary change: (1) the PTT-safety watchdog runs as its **own `_rx_watchdog` task** (only ever `await`s `sleep`, never the executor) so it drops PTT even while the decode loop is parked; (2) `_play_ambe` drops its frame if `mode != "rx"` so a resumed decode can't re-key a closed over; (3) `TxSession.close()` wraps the sign-off ID `transmit` so a raise can never skip the `ptt(False)` beneath it — hardening **every** streaming keyer; (4) `_force_unkey` drops `radio.ptt(False)` **directly** as a path-independent backstop. Verified on fakes that now model the failure (a `_BlockingVocoder` parking in the executor proves the over closes on its own with no teardown; a late decode proves no re-key; a spy radio raising on the sign-off transmit proves close() still unkeys); re-enable still gated on the dummy-load proof that surfaced it | Accepted |
| [0093](0093-aioc-panic-unkey.md) | **The AIOC transmitter can never be stranded keyed: drop the line first, unconditionally, atomically.** After ADR 0090/0091/0092 hardened the D-STAR *bridge*, the crossband STILL left the AIOC transmitting dead air on real hardware — only a process kill (closing the serial port) dropped PTT. A controlled **carrier key-test** (bare `pyserial` DTR toggle on a dummy load, operator watching, no audio/dongle/bridge) isolated it: **`dtr=False` with the port open cleanly unkeys this AIOC** — so the hardware is fine and the stuck-key is a SOFTWARE failure to reach `dtr=False`. Three `AiocBaofeng` bugs, all in the stranding class: (1) `ptt(False)` was guarded on `self._keyed`, so a desynced flag made the watchdog's / teardown's / REST `/ptt off`'s safety lever **no-op** (matches the journal: `/ptt off`→200, `status.transmitting`=False, carrier still up); (2) `_key_on` asserted the line THEN wrote the TX lead-in — a lead-in write that raised propagated with the line up but `_keyed` never set True, stranding it under (1); (3) `_key_off` **drained the stream before dropping the line**, so a `stop()` that blocks/raises on an xrun'd stream (the DV Dongle write-timeout wedge was starving it) kept the line asserted. Fix: a single unconditional `_drop_line()` primitive (bare `setattr(line,False)`, never guarded, can't block); `_key_off` drops the line FIRST then tears down the stream best-effort (the RF-safety inversion of ADR 0029's drain-then-drop — costs a few ms of tail clip); `_key_on` atomic (drop line on any post-assert failure); `ptt(False)` unconditional (guard removed). ## The live linking arc

Linking a repeater/base to a remote voice channel is done over **Mumble**, not the reverted M17 stack.
The arc is ADR **0041** (single link) → **0042** (multiple servers, DTMF-selectable, one active) →
**0043** (ungated disconnect) → **0045** (audio-correctness fixes) → **0049** (real-time DTMF mute +
Mumble→RF keying yield) → **0050** (the web UI doubles as a Mumble client). Config lives under
`[mumble]` / `[[mumble.servers]]`; see
[configuration.md](../configuration.md) and the annotated [`radio.toml.example`](../../radio.toml.example).
